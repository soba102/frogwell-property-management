問い合わせ対応の自動化 企画

仲介会社対応 ＆ テナント対応
既存の「請求書OCR・レポート・入金消込」プロトタイプに続く拡張テーマとして、 ① 仲介会社からの問い合わせ ② テナントからの問い合わせ の自動化アーキテクチャとフローを整理します。
Overview

2つの問い合わせ自動化

性質が異なる2系統です。仲介会社向けは「ナレッジを返す」自動応答型、 テナント向けは「内容を分類して関係者に振り分ける」トリアージ（仕分け）型です。 後者はテナント・スタッフ・業者・オーナーと関係者が多く、難易度は高めです。

① 仲介会社からの問い合わせ

  • 空き物件一覧の照会
  • 物件写真の依頼
  • 内見・鍵あけの手配

→ ナレッジ（空き状況・写真・ルール）を登録し、AIが照合して自動返答

② テナントからの問い合わせ

  • WhatsApp / LINE / メールで受付
  • 設備故障・修繕などを業者手配
  • オーナーへ費用負担を連絡・承認

→ AIが内容を分類し、ケース別に関係者へ振り分け・通知

Channel

受付チャネルは「共通モジュール」で設計（メールから検証）

まずは検証も兼ねて メールから開始し、受信部分を共通のアダプタ（差し替え可能な部品）として作ります。 AI分類やチケット管理といった中核の仕組みはそのままに、後から WhatsApp・LINE など手段を増やせる構造にします。 チャネルが変わっても運用フローは変わりません。

狙い： 中核（AI分類・チケット・通知）を一度作れば、チャネル追加は「アダプタを足すだけ」。 まずメールで仕組みを検証し、そのまま WhatsApp・LINE へ広げられます。

チャネル別の特徴

チャネル導入難易度特徴留意点
メール低 Gmail で即受付可能。送信の自由度が高い。 テナントがメールを使う前提。即時性はチャットに劣る。
LINE 公式アカウント中 Messaging API + Webhook で双方向対応。国内で普及。 友だち追加が前提。海外テナント比率が高いと不向き。
WhatsApp Business高 マレーシアで主流。テナント側の心理的ハードルが低い。 Meta の API 審査・テンプレート承認・24時間ルール・公式パートナー(BSP)経由が必要。
推奨アプローチ： 中核（AI分類・チケット管理）を共通モジュールとして先に固め、 まずメールで動作を検証。仕組みが固まったら WhatsApp を接続し、以降の追加（LINE 等）は アダプタを足すだけで対応します。メールは導入が容易で、API審査などの待ちもないため、検証の初手に最適です。
Reply Policy

自動返信のポリシー

誤回答を防ぐため、自動で返すのは「定型」だけに限定します。判断や費用が絡むものは、 必ず人が内容を確認してから返信する運用とし、AIは下書き作成までを担当します。

自動返信する（定型）

  • FAQ（ゴミ出し・設備の使い方・受付時間 等）
  • 受付完了の一次返信（「承りました」）
  • 空き状況・写真URLなど確定情報の案内

人が承認してから返信する

  • 修繕・業者手配など対応方針を伴うもの
  • 費用負担・見積・オーナー承認が絡むもの
  • 契約・退去・クレームなど判断が必要なもの
AIは「分類」「要約」「下書き作成」までを担い、最終送信の可否は人が握る ── という線引きにより、 省力化と安全性を両立します。運用に慣れてから、自動返信の対象を段階的に広げられます。
Roadmap

段階的な導入プラン

小さく始めて検証し、関係者連携・チャネルを段階的に広げます。

Phase 0

仲介会社向け：ナレッジ自動応答（クイックウィン）

メール受付 → AI分類 → 空き物件マスター/FAQ照合 → 返信文の自動生成（人が承認して送信）。鍵あけは担当へ通知。

最短で着手可能・既存スプレッドシート資産を活用
Phase 1

テナント向け：中核（AI分類＋チケット）を共通モジュールで構築

まずメール受付で、AIが分類・緊急度判定 → 問い合わせ管理表に起票 → 担当へ自動通知。FAQ・一次返信は自動、業者・オーナー連絡は「下書き提示→人が送信」。チャネルに依存しない中核をここで固める。

関係者連携を半自動で確立するフェーズ
Phase 2

WhatsApp 連携（共通アダプタで接続）

WhatsApp を共通チャネルアダプタ経由で接続し、双方向のやりとりに対応。Phase 1 の分類・チケットの仕組みをそのまま再利用。以降、LINE 等の追加もアダプタ追加だけで対応可能。

主要チャネル（WhatsApp）を本接続
Phase 3

業者手配・オーナー承認のワークフロー化

見積取得・オーナーの費用承認・日程調整を半自動化。確定費用を請求/レポート/消込に自動連携。自動返信の対象範囲を拡大。

運用が固まってから本格自動化
Integration

既存プロトタイプとの接続

今回の問い合わせ自動化は、既存の「OCR・レポート・消込」基盤とシームレスにつながります。

  • マスター再利用： 物件・オーナー・テナントマスターをそのまま参照し、問い合わせ主や物件を自動特定。
  • 修繕費の計上： 確定した修繕費を「請求書データ」に取り込み、オーナー別レポートの支出に自動反映。
  • 費用負担の整理： オーナー負担/テナント負担の区分を記録し、レポートや精算に連動。
  • 空き物件マスター： 仲介会社対応用の空き状況は、物件マスターを拡張して一元管理。
  • 業者マスター（新規）： 修繕業者の連絡先・得意分野・対応エリアを登録し、手配先の自動提案に利用。
Open Questions

ヒアリングで確認したい論点

アーキテクチャを確定するために、次の点を伺えると設計を具体化できます。

  1. テナントの主要連絡手段は？（WhatsApp 中心？ LINE？ メール？ 国・言語の比率も）
  2. 自動返信の範囲は？ 完全自動で返してよいか、人が承認してから送る運用にするか。
  3. 費用負担の判定基準は？ 契約上の負担区分でルール化できるか、都度オーナー承認か。
  4. 緊急時の対応体制は？ エスカレ先・対応時間・SLA（何分以内に一次対応 等）。
  5. 仲介会社向けの自動送信可否は？ 空き状況の更新は誰がどの頻度で行うか。
  6. 業者手配の現状フローは？ 既存の業者リスト・依頼方法・見積の取り方。
  7. 記録・個人情報の要件は？ やりとりの保存期間、アクセス権限、監査の要否。
上記が固まれば、Phase 0（仲介会社向けナレッジ応答）から実働モックを作り、 既存プロトタイプと同様に「動かして確認」しながら進められます。
