### 3.1 前提条件：テナント連絡先の対応表【着手前に必須】

現状の `マスター_テナント`（`TENANT_HEADERS`）には **メール・電話番号の列が無く**、差出人（メールアドレス / WhatsApp 番号）からテナント・物件・オーナーを引けない。**メール MVP（2-A）の着手前に**、テナントマスターへ次の列を追加する。

| 追加列 | 用途 |

| `メール` | メール差出人 → テナント特定 |

この対応表が無いと、問い合わせの送信者特定が機能しない（§5.7 のフォールバックに常時落ちる）。先方ヒアリングで **テナントの連絡先一覧**を入手し、本番マスターに反映する。

## 4. 共通アーキテクチャ

### 4.1 共通メッセージ形式（InboundMessage）

チャネル差を吸収する内部フォーマット。すべてのアダプタがこれに変換する。

```javascript
{
  id:           'msg_20260621_001',   // 内部 ID
  channel:      'email',              // email | whatsapp | line
  receivedAt:   '2026-06-21T10:30:00+08:00',
  fromAddress:  'tenant@example.com', // メール or 電話番号
  fromName:     'Tan Ah Kow',
  subject:      'Aircon not working', // WhatsApp では空 or 先頭行
  body:         '...原文...',
  attachments:  [{ name, mimeType, driveFileId }],
  threadId:     'gmail_thread_xxx',   // 返信スレッド用
  rawRef:       { gmailMessageId: '...' }
}
```

### 4.2 処理パイプライン

```
[チャネルアダプタ] → normalize → [AI分類] → [マスター照合] → [チケット起票/更新]
                                                      ↓
                                            [振り分け・通知]
                                                      ↓
                                            [返信ポリシー判定]
                                              ├ 自動送信（定型）
                                              └ 下書き → 人が承認 → 送信
```

Phase 1 の `Config.gs` にシート名を追加するか、`InquiryConfig.gs` で拡張する。

### 5.7 送信者・物件の特定とフォールバック【重要】

Phase 1 の請求書ですら、整形された帳票から物件を特定するのに住所・ファイル名・別名マスターを要した（HANDOFF §8.3）。**自由文のメール／チャットからの特定はさらに不安定**になる前提で設計する。

**特定の優先順位**

1. **差出人で引く（最優先・最も確実）**: メールアドレス / WhatsApp 番号 → `マスター_テナント` の連絡先列（§3.1）→ 物件・区画番号・オーナーが一意に決まる
2. **本文の手がかり**: AI 抽出の `property_hint` / `unit_hint` → `マスター_物件`・`マスター_建物別名` と照合
3. **どちらも一意に決まらない場合 → ステータス `要確認` で起票**（推測で確定しない）

**フォールバックの原則**

- 物件・テナントが一意に決まらないチケットは、**自動返信せず**「受付しました」一次返信のみ＋担当へ通知
- AI の `property_hint` 等は**あくまで候補**として別列に保持し、確定列（物件名・オーナー名）とは分ける
- Phase 1 の照合（建物＋ユニット → 単独一致 → 要確認）と同じ思想を踏襲する

## 6. メール MVP（Phase 2-A）— 最初に作るもの

Phase 1 と同じ **手動メニュー + 必要なら時間トリガー** で検証する。

> **検証チャネルは「仲介会社」から始める。** テナントは本番 WhatsApp 主体でメールをあまり使わないため、メールで検証するなら**メールが本命チャネルの仲介会社**が現実的（HTML 提案版の Phase 0 と整合）。  
> ただし作る中核（受信 → 分類 → チケット → 通知 → 返信ポリシー）は**チャネル非依存**で、後からテナント／WhatsApp にそのまま再利用する。

### 6.1 受信の取り方（Gmail）

**方式 A（推奨・MVP）: ラベル + 定期ポーリング**

1. 問い合わせ専用アドレスを用意（例: `inquiry@company.com` または Gmail エイリアス）
2. Gmail フィルタで `inquiry` ラベルを付与
3. GAS 時間トリガー（5〜15 分）または メニュー「問い合わせ受信チェック」で:
   - `GmailApp.search('label:inquiry is:unread')`
   - 各メッセージを InboundMessage に変換
   - 処理後 `message.markRead()` or 別ラベル `inquiry/processed`

**方式 B: Gmail 転送**

- テナント・仲介が使う実アドレス → 転送 → 監視ラベル  
  （本番 WhatsApp 前の**運用テスト**に向く）

**必要 OAuth スコープ（追加）**

```
https://www.googleapis.com/auth/gmail.readonly
または
https://www.googleapis.com/auth/gmail.modify
```

現状は `gmail.send` のみ → Phase 2 着手時に `appsscript.json` 更新が必要。

### 6.2 AI 分類（Gemini）

プロンプト出力を JSON 固定:

```json
{
  "category": "repair",
  "urgency": "medium",
  "property_hint": "The Mews",
  "unit_hint": "A-36-1",
  "tenant_hint": "Tan",
  "summary_ja": "エアコンが冷えないとの修繕依頼",
  "suggested_reply_ja": "お問い合わせありがとうございます。担当者が確認のうえ…",
  "auto_reply_ok": false
}
```

`InquiryMatcher.gs` の特定順は **①差出人（連絡先マスター）→ ②本文の `property_hint`/`unit_hint` → ③不明なら `要確認`**（詳細は §5.7）。AI の hint は候補列に入れ、確定列とは分離する。

### 6.3 通知（送信は Phase 1 と同じ Gmail）

| イベント | 通知先 | 内容 |
|----------|--------|------|
| 新規チケット（緊急） | `INQUIRY_NOTIFY_EMAIL` | 要約・物件・原文リンク |
| 新規チケット（通常） | 同上 or 分類別担当 | |
| 下書き作成完了 | 担当 | 承認用に下書き全文 |
| オーナー承認依頼（将来） | `マスター_オーナー.メール` | 見積・負担区分 |

### 6.4 スプレッドシート UI（メニュー案）

```
AI管理ツール
  …（Phase 1 既存）…
  ── 問い合わせ ──
  問い合わせ受信チェック（Gmail）
  未処理チケット一覧を更新
  選択チケット：下書き再生成
  選択チケット：返信送信（承認）
  選択チケット：完了にする
```

「選択」はアクティブ行 or プロンプトで受付 ID 入力（Phase 1 と同様の UX）。

### 6.5 MVP の完了定義（DoD）

2-A（仲介・メール）で中核が回ることを確認する。

- [ ] テスト用メール 1 通 → チケット起票 → 担当に通知
- [ ] 差出人 or 本文から物件が特定できる／できない場合は `要確認` に落ちる（§5.7）
- [ ] 空き照会・FAQ カテゴリ → 自動返信（設定 ON 時）
- [ ] 判断が要るカテゴリ（修繕等）→ 下書きのみ、人がボタンで送信
- [ ] 返信が **同じ Gmail スレッド** に返る（`GmailThread.reply()`）
- [ ] 履歴シートに受信・送信が残る

## 7. 自動返信ポリシー（具体ルール）

| 分類 | 自動送信 | 人の承認 | 備考 |
|------|:--------:|:--------:|------|
| FAQ / 一般 | ◎ | | ナレッジ一致度が高い場合のみ |
| 受付完了（一次返信） | ◎ | | 「承りました。担当が確認します」 |
| 空き照会・写真 URL | △ | ◎ 推奨 | マスター鮮度が重要 |
| 修繕・設備 | | ◎ | 業者・オーナー連携前 |
| 緊急 | | ◎ | 自動は一次返信のみ。本体対応は人 |
| 契約・退去・クレーム | | ◎ | |
| 支払い・賃料 | | ◎ | 消込シート参照の下書きは可 |

**原則**: AI は **分類・要約・下書き** まで。最終送信はポリシー表 + `INQUIRY_AUTO_REPLY` で制御。

## 8. WhatsApp 接続（Phase 2-C）— テナント本番チャネル

テナントの本命チャネル。**メールで固めた中核（InboundMessage → 分類 → チケット → 通知 → 返信ポリシー）には一切手を入れず**、入口（受信）と出口（送信）のアダプタだけを足す。

### 8.0 まず理解すべき「WhatsApp はメールと別物」な点

メール送信（Phase 1）の感覚で実装すると詰まる。先に押さえるべき差分：

| 論点 | メール | WhatsApp Business |
|------|--------|-------------------|
| 送信のきっかけ | いつでも自由 | **24h ルール**（下記）で制限 |
| こちら起点の連絡 | 自由文で可 | **事前承認テンプレート**が必須 |
| 受信方法 | Gmail をポーリング | **Webhook（push）** をサーバーで受ける |
| 個人アカウント | 不要 | **ビジネス番号の登録・認証**が必要 |
| 費用 | 実質無料 | **会話単位の従量課金** |
| 宛先キー | メールアドレス | **国際電話番号**（`+60...`） |

### 8.1 接続方式の選定

| 方式 | 概要 | メリット | デメリット | 向くケース |
|------|------|----------|------------|-----------|
| **Meta Cloud API 直結** | Meta 公式の Cloud API を直接叩く | 中間コストなし・公式 | 自前で Webhook サーバー・審査対応 | エンジニアが API を扱える |
| **BSP 経由（Twilio / 360dialog 等）** | 公式パートナー経由 | SDK・サンプル豊富、番号調達が楽 | 月額＋従量、ベンダーロックイン | 早く確実に立ち上げたい |

**推奨**: 検証は **BSP（Twilio 等）** で素早く、運用コストが見えてから直結を検討。

### 8.2 受信（Webhook）の置き場所 ── GAS の限界

WhatsApp は **Webhook で push** されるため、Gmail のようなポーリングはできない。

- **GAS の `doPost`（Web アプリ公開）でも受信は可能**だが、実行時間・同時実行・再試行制御に弱く、本番には不安。
- **推奨構成**: `Cloud Functions / Cloud Run` で Webhook を受け、(a) スプレッドシートへ直接行追記、または (b) GAS の Web アプリへ転送。以降の分類・チケット・通知は既存 GAS をそのまま使う。

```
[テナント] → WhatsApp → [Meta/BSP] --Webhook--> [Cloud Functions]
                                                   → Sheets に受信行追記
                                                   → GAS（分類→チケット→通知）をトリガー
```

### 8.3 送信と「24 時間ルール」

- **24h ウィンドウ**: テナントから最後のメッセージを受信してから **24 時間以内**は自由文で返信可（＝問い合わせ対応はほぼこの中で完結する）。
- **24h を超える／こちら起点**（例: 「業者の訪問日が決まりました」を後日送る）は、**Meta 承認済みテンプレート**でしか送れない。
- → よく使う通知（受付完了・訪問日連絡・完了報告）は**テンプレートを事前申請**しておく。
- **オプトイン**: ビジネスメッセージ受信の同意取得が前提。

### 8.4 マスター・設定の追加

- `マスター_テナント` に `WhatsApp番号`（`+60...`）列 ← §3.1 で追加済みの連絡先列を流用
- `設定` シート: `WHATSAPP_API_TOKEN` / `WHATSAPP_PHONE_NUMBER_ID`（§5.6）
- Webhook 検証トークン等は Cloud Functions 側の環境変数で保持（スプレッドシートに置かない）

### 8.5 アダプタの責務（中核は不変）

受信側:

```
WhatsApp Webhook
  → parse payload（送信元番号・本文・添付メディアID）
  → メディアは Meta/BSP から取得して Drive 保存
  → InboundMessage { channel: 'whatsapp', fromAddress: '+60...', ... }
  → 既存パイプライン（分類→照合→チケット→通知）へ ← ここから先はメールと共通
```

送信側:

```
InquiryReply.send(ticket, body)
  → if channel === 'email'    → GmailThread.reply()
  → if channel === 'whatsapp' →
       24h 以内 ? 自由文メッセージ
                : 承認済みテンプレート（変数を差し込み）
```

| **2-A** | **仲介向け**：メール受付 + 分類 + チケット + 通知 + 空き/FAQ 自動応答（＝チャネル非依存の**中核**を仲介メールで検証） | メール | GAS モジュール、新シート、空き・FAQ マスター | Phase 1 モック、Gmail 読取 scope |

| **2-C** | **WhatsApp アダプタ**：テナントの本番チャネルを接続 | WhatsApp | Webhook、番号マスター、承認テンプレ | 2-B 完了、Meta 審査 |
| **2-D** | 業者手配・オーナー承認・修繕費計上 | - | ワークフロー列、請求書データ連携 | 2-B 以降、先方フロー確認 |

**着手順の推奨**

1. **2-A（仲介・メール）** でチャネル非依存の中核を固める＝クイックウィン（仲介はメールが本命なので検証と本番が一致）
2. **2-B（テナント・分類トリアージ）** を中核に積み増し（連絡先マスターが前提・§3.1）
3. 先方ヒアリング（§10）が返ったら **2-C（WhatsApp）** の API 選定・接続
